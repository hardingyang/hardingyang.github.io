Session_Cookie_Token总结

Jenkooo
2020-04-22
总结 › Session_Cookie_Token总结
总结

Session

Session表示的是一次会话，不同的人与同一个服务器的对话就会产生不同的SessionID，由于Web是无状态的请求，所以在Web后期产生的问题也更加明显，比如服务器如何知道你是谁，哪个人登录了系统等等，所以在初期，人们就想到了用SessionId作为一个认证的标志，SessionId是一个随机的字符串，每个人都不同，客户端将SessionID发给了服务器，服务器将SessionId存在数据库，并记录这对应的是谁，那么每次人的请求都需要查一次数据库，当人数直线上升后，服务器的压力就开始加大，并且当服务器产生集群后，那么SessionID的表也需要赋值到多个机器上，这样的做法扩展性十分的差，然后也有人建议把SessionID的查询单独放到一个服务器中，但是当这个服务器宕机后，也会出现问题，所以这是很不理智的行为，所以随着Web的发展，开始产生了Token

Token

Token表示一个令牌，是为了解决SessionID认证中产生的服务器压力而产生的解决方式，Token由三个部分组成：头部、有效负载、签名

三个组成部分通过加密算法如HS256算法，然后产生了一个很长的字符串。解决SessionID中产生服务器压力的过程是：

  1. 客户端将username,password发给服务器
  2. 服务器将username,password放到payload中然后通过Token的算法产生一个字符串
  3. 将token字符串发回给客户端
  4. 客户端存储token在客户那里(如何存？)
  5. 每次有新的带身份的请求就将这个存储的token放在请求的header中发回给服务器
  6. 服务器检查header，然后将发回来的token字符串用Token的秘钥解开，看是否正确，不能正确解密那么就会在服务器端报错，那么就是被人篡改的token或者不是本人
  7. 如果token验证通过，那么就可以确定现在是你本人，那么服务器就正常返回客户端所需要的东西

Cookie

Cookie是一种客户端保留数据的一种功能，Cookie由服务器生成，发送给浏览器，浏览器以key：value的形式保存在某个目录下的文本文件中，下一次请求同一个网站的时候就会在Cookie中带上这个key：value发送给服务器，然后服务器就恶意得到浏览器发送过来的Cookie，同时Cookie也存在一些限制，比如存活时间和大小和Cookie的个数，如果在服务器中没有设置Cookie的存活时间，那么Cookie的存活时间就是浏览器的关闭，浏览器关闭之后Cookie就不复存在
